# 按需定制是如何工作的

## 心理模型

将 `resume.md` 想象成一个包含你所做过一切的数据库。每次工作申请就是对这个数据库的一次查询。

```
resume.md       (你所有的职业数据)
  × tags        (WHERE — 过滤相关内容)
  × hide/pin    (SELECT — 选择哪些部分，以及按什么顺序)
  × vars        (SET — 针对特定职位的关键词、标语)
  × style       (FORMAT — 展现形式的调整)
  → output      (PDF、HTML、DOCX 或 PNG)
```

你的职业数据是持久的，这是多年积累的结果。但是对于每一次申请，呈现数据的方式都会发生变化。标签 (Tags) 过滤内容，视图 (Views) 塑造输出结果。

## 标签与视图 (Tags and Views)

两个独立的概念结合起来产生了最终的每次渲染：

|                      | 目的         | SQL 类比            |
| -------------------- | ------------ | ------------------- |
| [标签](/guide/tags)  | 过滤内容     | `WHERE`             |
| [视图](/guide/views) | 配置渲染规则 | `SELECT` + `FORMAT` |

**标签**决定了哪些内容保留，哪些剔除。将 `{.@frontend}` 添加到一个列表项上，它只会在为 `frontend` 渲染时出现。你可以组合标签（例如 `fullstack: [frontend, backend]`）来构建可复用的内容集。标签纯粹是过滤器，它们从不涉及渲染过程。

**视图**决定了过滤后的内容如何渲染：哪些部分出现，以什么顺序排列，注入哪些关键词，目标页数是多少。每次渲染都会使用视图，即使你从未显式创建一个视图。

这两者都是可选且独立的。你可以只使用标签而不用视图，或者只用视图而不用标签，或者两者结合使用。一份既没有标签又没有视图配置的简历将生成一个包含所有内容的单一 PDF 文件。
