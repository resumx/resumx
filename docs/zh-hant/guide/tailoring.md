# 客製化工作原理 (How Tailoring Works)

## 思維模型

把 `resume.md` 想像成您所有經歷的資料庫。每一次申請工作就像是對這個資料庫進行一次查詢。

```
resume.md       (您的所有職涯資料)
  × tags        (WHERE — 過濾出相關內容)
  × hide/pin    (SELECT — 要顯示哪些區塊，順序為何)
  × vars        (SET — 針對特定職位的關鍵字、標語)
  × style       (FORMAT — 呈現樣式調整)
  → output      (輸出為 PDF、HTML、DOCX 或 PNG)
```

您的職涯資料是持久的，它會隨著時間累積。但您呈現它的方式會因為每次申請而改變。標籤 (tags) 過濾內容，視圖 (views) 塑造輸出。

## 標籤與視圖 (Tags and Views)

兩個獨立的關注點結合起來產生每次的渲染結果：

|                              | 目的         | SQL 類比            |
| ---------------------------- | ------------ | ------------------- |
| [標籤 (Tags)](/guide/tags)   | 過濾內容     | `WHERE`             |
| [視圖 (Views)](/guide/views) | 配置渲染方式 | `SELECT` + `FORMAT` |

**標籤** 決定什麼內容要保留，什麼要排除。如果在一個項目符號中加上 `{.@frontend}`，那麼它只會在針對 `frontend` 進行渲染時出現。組合標籤（`fullstack: [frontend, backend]`）可以建立可重複使用的內容集。標籤純粹是過濾器，它們絕不會干涉渲染。

**視圖** 決定被過濾的內容如何渲染：哪些區塊會出現、排列順序為何、注入哪些關鍵字、要調整的目標頁數是多少。每次渲染都會使用一個視圖，即使您從未明確建立過。

兩者是獨立且可選的。您可以只用標籤不用視圖，只用視圖不用標籤，或者兩者一起使用。如果兩者都不用，履歷會產生一份包含所有內容的單一 PDF。
