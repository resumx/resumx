# 單欄與雙欄

雙欄履歷看起來很時尚。但數據表明它們會讓你失去面試機會。

## 招募人員實際上是如何閱讀履歷的

[TheLadders 眼動追蹤研究](https://www.theladders.com/career-advice/why-do-recruiters-spend-only-7-4-seconds-on-resumes)（2012 年，更新於 2018 年）使用眼動追蹤硬體在 10 週內追蹤了 30 位專業招募人員。他們關於版面的主要發現：

- 招募人員在做出合適/不合適的決定之前，平均花費 **7.4 秒**進行初步掃描。
- 眼睛遵循 **F 形模式**：從左上角開始，向右掃描，然後沿著左邊緣向下移動並再次向右掃描。
- 雙欄版面被確定為一個**負面的設計元素**，被認為是「雜亂的」，這打斷了自然的掃描流暢度。
- 為了應對這種雜亂，招募人員會集中在頁面上**較少的區域**，這意味著他們閱讀你內容的*數量*比在單欄履歷上還要少。

單欄版面自然地符合 F 形模式。雙欄迫使眼睛選擇要遵循哪一欄，從而打破了掃描的過程。

### 對眼動追蹤研究的保留意見

這項研究是關於招募人員閱讀行為被引用最廣泛的來源，但它有實質的局限性：

- **30 名招募人員**是一個極小的樣本。當時美國大約有 39,000 家人事機構。
- 方法論從未經過同儕審查。招募人員是如何被計時的、使用了哪種眼動追蹤硬體，以及每次測試審閱了多少份履歷都沒有公開。
- TheLadders 是一個求職網站，而不是一個研究機構。「7.4 秒」這個數字起到了行銷內容的作用。
- 該研究並沒有將欄位版面這個變數與其他設計變數（字體選擇、密度、空白）分開。關於「雜亂」的發現可能反映的是所測試的特定模板，而不是一般情況下的雙欄版面。

儘管有這些局限性，F 形模式的發現與幾十年來關於人類如何掃描文件和網頁的更廣泛 UX 眼動追蹤研究是一致的。

## ATS 系統如何解析欄位

認為「現代 ATS 什麼都能讀」的觀念是錯誤的。雖然像 Greenhouse 和 Lever 這樣的頂級系統正在改進，但大多數 ATS 市場仍然使用簡單的線性文字萃取。

### 解析機制

大多數 ATS 解析器**嚴格地由左至右、由上至下**閱讀文件。當它們遇到雙欄版面時，它們不會先處理 A 欄再處理 B 欄。它們會讀取左欄的第一行，然後緊接著讀取右欄的第一行，將不相關的內容混在一起。

如果你的左欄有「Skills（技能）」，而你的右欄有「Experience（經驗）」，ATS 的輸出可能看起來像這樣：

```
Python  Senior Software Engineer
React   Acme Corp, 2021-2024
SQL     Led migration of payment service...
```

這在招募人員的後台儀表板中產生了無法閱讀的亂碼。你的履歷並沒有「被拒絕」，而是因為解析出的欄位變成亂碼，在關鍵字搜尋中變得**隱形**。

### Enhancv ATS 解析研究 (2023)

[Enhancv](https://enhancv.com/blog/busting-ats-myths/) 測試了來自 Microsoft Office、Canva、Google Docs 以及他們自己建立器中的數十種履歷模板，對抗 Indeed 的 ATS，並對每個部分被解析的準確度進行評分。

**整體平均：**

| 版面 | 平均解析分數 |
| ---- | ------------ |
| 單欄 | 93%          |
| 雙欄 | 86%          |

**雙欄履歷流失最多的地方：**

| 區塊                  | 單欄  | 雙欄 |
| --------------------- | ----- | ---- |
| 技能 (Skills)         | 65%   | 46%  |
| 教育 (Education)      | ~100% | 88%  |
| 證照 (Certifications) | ~100% | 86%  |
| LinkedIn / 社群連結   | 100%  | 82%  |

技能解析下降了幾乎 20 個百分點，而這正是招募人員最常過濾的區塊。

**「構建良好的工具」例外**

Enhancv 也進行了一項過濾分析，只看 Google Docs 和他們自己的產品（排除 Canva 和 MS Office）。在這個子集中，雙欄模板的分數實際上**高於** (98%) 單欄 (95%)。這表明欄位是如何構建的（語義結構與文字方塊）比欄數本身更重要。

**對 Enhancv 研究的保留意見**

在相信 98% 的雙欄分數之前，考慮一下這項研究的局限性：

- **小樣本：** 98% 的數字來自過濾後的子集，原始文章也承認它只適用於「某些情況下」。
- **單一 ATS：** 僅針對 Indeed 進行測試。在像 Workday 或 Taleo 這樣更嚴格的系統上，結果會有所不同。
- **整體的真相：** 當包含所有工具時，**除了地點標題外，單欄在每個區塊的解析都更好**。
- **技術不透明：** 該研究沒有公開具體測試的模板，這使得很難獨立驗證是什麼讓一個模板「構建良好」。

### 特定 ATS 平台如何處理欄位

不同的系統有不同的失敗方式。[Jobscan 向 Lever 提交了一份雙欄履歷](https://www.jobscan.co/blog/resume-tables-columns-ats/)並公佈了解析輸出來作為證明：ATS 成功提取了工作經驗，但**完全忽略了第二欄**。申請人的技能、摘要和聯絡資訊都消失了。

雖然沒有任何系統能完美處理欄位，但行為因供應商而異：

- **Taleo (Oracle)：** 嚴格從左至右讀取。雙欄會導致解析器在區塊之間隨機跳躍，打亂內容 ([Hireflow](https://hireflow.net/blog/taleo-resume-parsing-problems-explained))。
- **iCIMS：** 經常完全跳過放在表格或欄位中的內容 ([Princive](https://www.princive.com/ats/systems/icims))。
- **Workday：** 非常嚴格。除非使用原生 Microsoft Word 的欄位格式（絕對不是文字方塊）來建立，否則雙欄將會失敗 ([Princive](https://www.princive.com/ats/systems/workday))。
- **Greenhouse / Lever：** 比舊系統更寬容，但對於複雜的版面仍然不可靠 ([AI ResumeGuru](https://airesume.guru/blog/single-column-vs-two-column-resume-which-layout-gets-more-interviews/))。

### Canva 與設計工具匯出

從 Canva 或 Figma 等設計工具匯出的履歷值得特別一提。這些工具生成 PDF 時使用絕對定位，將每個文字區塊放在頁面上的精確座標處。解析器沒有語義閱讀順序可以遵循。行業估計，對於較舊的 ATS 系統，設計工具匯出的解析成功率**低於 5%**。即使在現代系統上，結果也是不可靠的。

## 建議

**當透過線上入口網站提交申請時，我們建議使用單欄版面。** 理由很簡單：單欄設計可以最大程度地降低解析錯誤的風險，確保你的資訊能被幾乎所有 Applicant Tracking Systems 準確讀取。

然而，**雙欄版面仍然有其用武之地。它們非常適合直接溝通，例如發電子郵件給招聘經理、在社交活動中分發紙本複印件，或在你的個人網站上分享。** 在這些情況下，雙欄設計提供的視覺吸引力和空間的高效利用可以非常有效。
